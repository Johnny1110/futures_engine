# 從零打造 Crypto Perpetual Futures Exchange — Golang 架構師路線圖

## 系統全貌：一個永續合約交易所裡到底在跑什麼？

在你寫第一行 code 之前，先理解整個系統的 data flow。當一個 trader 按下「Market Buy 10 BTC-USDT Perp」，背後發生的事情：

```
Client (WebSocket) 
  → API Gateway (認證、限流、參數驗證)
    → Order Router (風控前置檢查)
      → Risk Engine (保證金檢查、倉位限額)
        → Matching Engine (撮合核心，純記憶體運算)
          → Trade 產生
            ├→ Execution Report → 回推給 Client
            ├→ Position Engine (更新倉位、計算未實現 PnL)
            ├→ Settlement Engine (結算已實現 PnL、手續費)
            ├→ Market Data Publisher (推 K-line、BBO、depth)
            └→ Append-Only Trade Log (持久化、可重播)
```

**關鍵設計哲學**：Matching Engine 是整個系統的「心臟」，它必須是 **deterministic（確定性的）** 且 **single-threaded**。為什麼？因為撮合邏輯不能有任何 race condition，而且當你需要從 log 重播恢復狀態時，同樣的 input 必須產生完全相同的 output。這就是為什麼所有真實交易所的撮合引擎都是 single-writer model。

## 開發階段 Roadmap

### Phase 1：Order Book — 交易所的骨架
**目標**：理解 limit order book 的資料結構與行為

**金融知識**：
- **Limit Order vs Market Order**：Limit order 是「我要在這個價格買/賣」，它提供流動性（maker）。Market order 是「我要用現在最好的價格立刻成交」，它消耗流動性（taker）。
- **Bid/Ask/Spread**：Bid 是所有買單中最高價，Ask 是所有賣單中最低價。Spread = Ask - Bid，這是 market maker 的利潤來源。
- **Price-Time Priority (FIFO)**：同價位的訂單，先到的先成交。這是最基本也最常見的撮合規則。

**核心資料結構**：

```go
// Order 代表一張訂單，這裡的設計重點是：
// 1. 用 decimal 而非 float64（精度問題，後面會解釋）
// 2. RemainingQty 與 Qty 分離，因為一張訂單可能被部分成交多次
type Side int
const (
    Buy  Side = iota
    Sell
)

type Order struct {
    ID           uint64
    Side         Side
    Price        decimal.Decimal  // github.com/shopspring/decimal
    Qty          decimal.Decimal  // 原始數量
    RemainingQty decimal.Decimal  // 剩餘未成交數量
    Timestamp    int64            // UnixNano，用於 FIFO 排序
}

// PriceLevel 代表同一價位的所有訂單
// 為什麼用 linked list？因為 FIFO 需要頻繁的頭部移除和尾部插入，
// 這正是 list 的 O(1) 強項。Array 的頭部移除是 O(n)。
type PriceLevel struct {
    Price  decimal.Decimal
    Orders *list.List  // container/list，doubly linked list
    Total  decimal.Decimal  // 這個價位的總數量（快速查詢 depth）
}

// OrderBook 的核心：兩棵樹
// 為什麼用 Red-Black Tree (或 BTree)？
// 因為撮合時需要快速找到「最佳價位」：
// - Buy side：最高價（樹的 max）
// - Sell side：最低價（樹的 min）
// 同時還需要 O(log n) 的插入和刪除
type OrderBook struct {
    Symbol string
    Bids   *rbtree.Tree  // 按價格降序（最高買價在根附近）
    Asks   *rbtree.Tree  // 按價格升序（最低賣價在根附近）
    Orders map[uint64]*Order  // Order ID → Order，O(1) 查找用於取消
}
```

**為什麼不用 `float64`？** 這是所有金融系統的第一課。`0.1 + 0.2 = 0.30000000000000004` 在 IEEE 754 中。如果你的撮合引擎用 float64，累積下來帳會對不上。解法有兩種：用 `decimal` 庫（如 shopspring/decimal），或者用 `int64` 表示最小單位（例如 1 satoshi = 1, 1 BTC = 100_000_000）。後者性能更好，真實交易所多用此方式。

**演算法：撮合邏輯**

```
func (ob *OrderBook) PlaceLimitOrder(order *Order) []Trade:
    if order is Buy:
        while order.RemainingQty > 0 AND asks is not empty:
            bestAsk = asks.Min()
            if order.Price >= bestAsk.Price:
                match(order, bestAsk) → 產生 Trade
            else:
                break  // 價格不夠好，無法成交
        if order.RemainingQty > 0:
            insert order into bids  // 剩餘部分掛單
    // Sell side 鏡像對稱
```

**實作任務**：
1. 實作 `OrderBook` 結構（可先用 sorted slice，再換 rbtree）
2. 實作 `PlaceLimitOrder`、`PlaceMarketOrder`、`CancelOrder`
3. 寫 benchmark：10 萬筆 order 的撮合吞吐量
4. 輸出 L2 depth snapshot（每個價位的總量）

### Phase 2：Matching Engine — 從 Order Book 到撮合引擎
**目標**：將 Order Book 包裝成一個 event-driven 的撮合引擎

**系統知識**：

**Event Sourcing 模式**：
傳統 CRUD 系統直接修改狀態。Event sourcing 不一樣 — 它把「發生了什麼事」記錄下來，狀態是從 event 推導出來的。在交易所中：

```
Events (不可變的事實紀錄):
  OrderPlaced{ID:1, Buy, Price:50000, Qty:1}
  OrderPlaced{ID:2, Sell, Price:50000, Qty:0.5}
  TradeExecuted{BuyID:1, SellID:2, Price:50000, Qty:0.5}
  OrderPlaced{ID:3, Sell, Price:50000, Qty:0.5}
  TradeExecuted{BuyID:1, SellID:3, Price:50000, Qty:0.5}

State (從 events 推導):
  Order 1: Filled
  Order 2: Filled
  Order 3: Filled
```

**為什麼交易所要用 event sourcing？**
1. **Auditability**：監管機構要求你能重播任何一筆交易的完整歷史
2. **Recovery**：引擎崩潰後，從最後一個 snapshot + 後續 events 重播即可恢復
3. **Debugging**：線上出現詭異的成交？重播那段時間的 events 就能復現

**Sequencer + Single Writer 模式**：

```go
// Sequencer 是撮合引擎的入口，負責：
// 1. 將所有 inbound message 排成全局順序
// 2. 分配 sequence number（單調遞增）
// 3. 保證同一時間只有一個 goroutine 在處理撮合
type Sequencer struct {
    inbound  chan OrderCommand      // 所有 order 請求進入此 channel
    sequence uint64
    engine   *MatchingEngine
}

// OrderCommand 是命令（意圖），不是事件（事實）
type OrderCommand struct {
    Type      CommandType  // PlaceOrder, CancelOrder, AmendOrder
    Order     Order
    ResponseCh chan<- ExecutionReport  // 回傳結果給 API layer
}

// ExecutionReport 是撮合結果
type ExecutionReport struct {
    OrderID    uint64
    Status     OrderStatus  // New, PartiallyFilled, Filled, Cancelled, Rejected
    Trades     []Trade
    RejectCode RejectReason
}

// 核心 loop — 這就是 single writer
func (s *Sequencer) Run() {
    for cmd := range s.inbound {
        cmd.Sequence = atomic.AddUint64(&s.sequence, 1)
        report := s.engine.Process(cmd)
        s.appendToLog(cmd, report)  // append-only log
        cmd.ResponseCh <- report
    }
}
```

**Append-Only Log 的設計**：

為什麼不直接寫 database？因為撮合引擎的吞吐量可能是每秒數十萬筆，而 DB write 的延遲太高。真實做法是：
1. 撮合引擎 → append-only binary log（本地文件，sequential write，極快）
2. 另一個 goroutine 異步消費 log → 寫入 DB / Kafka / 推送 market data

這就是 LMAX Disruptor 的核心思想：用 mechanical sympathy（對硬體友善的設計）來達到極低延遲。

**實作任務**：
1. 實作 `Sequencer`，用 single goroutine 消費 channel
2. 實作 binary log writer（先用 `encoding/gob`，後續可換 flatbuffers）
3. 實作 log replay：從 log 重建 order book 狀態
4. 驗證 determinism：replay 後的狀態必須與原始狀態完全一致

### Phase 3：Position & Margin — 永續合約的靈魂
**目標**：理解永續合約特有的保證金、倉位、清算機制

**金融知識**：

**什麼是永續合約？** 它是一種「永不到期」的期貨合約。傳統期貨有交割日（例如 BTC-0630），永續合約沒有。那問題來了：如果沒有到期日，合約價格要怎麼錨定現貨價格？答案就是 **Funding Rate**。

**Funding Rate 機制**：
每 8 小時結算一次。如果合約價格 > 現貨價格（代表多頭過多），多頭付錢給空頭。反之亦然。這個機制讓合約價格不斷被拉回現貨價格附近。

```
Funding Payment = Position Size × Funding Rate
如果 Funding Rate = +0.01%，你持有 10 BTC 多倉：
你要支付 10 × 0.01% = 0.001 BTC 給空頭持倉者
```

**保證金模式**：
- **Isolated Margin**：每個倉位獨立計算保證金。爆倉只影響該倉位。
- **Cross Margin**：所有倉位共享帳戶餘額。一個倉位的未實現利潤可以幫助另一個倉位避免爆倉。

**Mark Price vs Last Price**：
為什麼不能用最新成交價來計算倉位盈虧？因為有人可能用一筆小單把價格打到極端位置，觸發大量清算。所以交易所會用 **Mark Price**（通常是多個交易所的加權平均現貨價格 + 基差），它更穩定，不容易被操縱。

```go
type Position struct {
    Symbol        string
    Side          Side
    Size          decimal.Decimal  // 倉位大小（合約張數或幣數）
    EntryPrice    decimal.Decimal  // 平均開倉價
    Leverage      int              // 槓桿倍數
    MarginMode    MarginMode       // Isolated / Cross
    Margin        decimal.Decimal  // 已分配保證金
    
    // 以下為計算欄位
    UnrealizedPnL decimal.Decimal
    LiqPrice      decimal.Decimal  // 清算價格
}

// 未實現盈虧計算
// Long:  PnL = Size × (MarkPrice - EntryPrice)
// Short: PnL = Size × (EntryPrice - MarkPrice)
func (p *Position) CalcUnrealizedPnL(markPrice decimal.Decimal) decimal.Decimal {
    diff := markPrice.Sub(p.EntryPrice)
    if p.Side == Sell {
        diff = diff.Neg()
    }
    return p.Size.Mul(diff)
}

// 清算價格計算（簡化版，忽略手續費和 funding）
// Long:  LiqPrice = EntryPrice × (1 - 1/Leverage + MaintenanceMarginRate)
// Short: LiqPrice = EntryPrice × (1 + 1/Leverage - MaintenanceMarginRate)
// 真實計算更複雜，需要考慮 maintenance margin ratio tiers
func (p *Position) CalcLiquidationPrice(mmRate decimal.Decimal) decimal.Decimal {
    one := decimal.NewFromInt(1)
    leverageInv := one.Div(decimal.NewFromInt(int64(p.Leverage)))
    if p.Side == Buy {
        factor := one.Sub(leverageInv).Add(mmRate)
        return p.EntryPrice.Mul(factor)
    }
    factor := one.Add(leverageInv).Sub(mmRate)
    return p.EntryPrice.Mul(factor)
}

// Maintenance Margin Rate 分層（Binance 風格）
// 倉位越大，維持保證金率越高 — 這是為了降低大倉位的系統性風險
type MarginTier struct {
    MaxNotional        decimal.Decimal  // 名義價值上限
    MaintenanceRate    decimal.Decimal  // 維持保證金率
    MaintenancePremium decimal.Decimal  // 保證金累計調整值
}
```

**實作任務**：
1. 實作 `PositionManager`：開倉、加倉、減倉、平倉
2. 加倉時的平均價計算：`newEntry = (oldSize × oldEntry + addSize × addPrice) / (oldSize + addSize)`
3. 實作 `MarginCalculator`：isolated / cross margin 計算
4. 實作清算價格計算（含 tiered maintenance margin）
5. 實作 Mark Price 模擬（先用簡單的 EMA，之後可接真實數據）

### Phase 4：Risk Engine & Liquidation — 保護交易所不破產
**目標**：實作風險控制和清算機制

**系統知識**：

**為什麼 Risk Engine 必須在撮合之前？** 如果一個 trader 帳戶餘額只剩 100 USDT，他試圖開一個 100x leverage 的 1 BTC 倉位，你不能讓這筆訂單進入 matching engine。Risk Engine 是 gatekeeper。

**清算引擎的觸發流程**：

```
Mark Price 更新
  → 遍歷所有 position
    → 計算每個 position 的 margin ratio
      → margin ratio = maintenance margin / (margin + unrealized PnL)
        → if margin ratio >= 100%:
          → 觸發清算
            → 步驟 1：嘗試市場清算（發 reduce-only market order）
            → 步驟 2：如果市場流動性不足 → Insurance Fund 介入
            → 步驟 3：如果 Insurance Fund 不足 → ADL (Auto-Deleveraging)
```

```go
// RiskEngine 在每個 order 進入前做檢查
type RiskEngine struct {

    balances   *BalanceManager
    marginTier []MarginTier
```

**Insurance Fund 的角色**：
清算發生時，如果能在 bankruptcy price（倉位價值歸零的價格）之前成交，差額歸入 Insurance Fund。如果清算單無法在 bankruptcy price 之前成交（即產生穿倉），Insurance Fund 彌補損失。這就是為什麼 Binance 的 Insurance Fund 有數億美金。

### Phase 5：Funding Rate & Settlement
**目標**：實作永續合約的 funding rate 計算與結算

**金融知識（深入版）**：

```
Funding Rate = clamp(Premium Index + Interest Rate, -0.75%, +0.75%)

Premium Index = ( Impact Bid Price + Impact Ask Price ) / 2 - Spot Index Price
                ─────────────────────────────────────────────────────────────
                                    Spot Index Price

Impact Bid/Ask Price：
  不是 best bid/ask，而是「能消耗 Impact Notional 的數量加權平均價」
  這能抵抗小額掛單的操縱

Interest Rate：通常固定 0.03%/day (0.01% per 8h)
```

```go
    impactNotional decimal.Decimal  // 例如 200 USDT
    interestRate   decimal.Decimal  // 通常 0.01%
    clampMax       decimal.Decimal  // +0.75%
    clampMin       decimal.Decimal  // -0.75%
}

// 計算 Impact Price：模擬一筆 impactNotional 的 market order 在 order book 上的成交均價
func (f *FundingCalculator) CalcImpactPrice(book *OrderBook, side Side) decimal.Decimal {
    remaining := f.impactNotional
    totalQty := decimal.Zero
    for level := book.BestPrice(side); remaining.IsPositive(); level = book.NextLevel(side) {
        fillQty := decimal.Min(remaining.Div(level.Price), level.Total)
        totalQty = totalQty.Add(fillQty)
        remaining = remaining.Sub(fillQty.Mul(level.Price))
    }
    return f.impactNotional.Div(totalQty)
}
```

### Phase 6：API Layer & WebSocket — 讓外部世界接入
**目標**：建立 REST + WebSocket 接口

**系統知識**：

**為什麼交易所需要 WebSocket？**
HTTP 是 request-response 模式，不適合推送即時行情。WebSocket 是全雙工的，server 可以主動推 price update、order update、position update。但管理數萬甚至數百萬的 WebSocket 連線是一個重大挑戰。

**API 設計原則**：
- REST 用於：下單、查詢餘額、查詢歷史（低頻、需要可靠回應）
- WebSocket 用於：即時行情、order update、position update（高頻、允許偶爾丟失）

### Phase 7：Persistence & Recovery — 不能丟資料
**目標**：實作資料持久化和系統恢復

**系統知識**：

**WAL (Write-Ahead Log) 模式**：
這是所有需要 durability 的系統的核心模式（PostgreSQL 用它，Kafka 也是）。原則很簡單：在修改內存狀態之前，先把操作寫入 log。

```
寫入流程：
1. Serialize command → binary
2. Append to WAL file (fsync)
3. Apply to in-memory state
4. Return success

恢復流程：
1. Load latest snapshot (if exists)
2. Replay WAL entries after snapshot
3. In-memory state fully restored
```

```go
// WAL Entry 設計
type WALEntry struct {
    Sequence  uint64
    Timestamp int64
    Type      EntryType
    Payload   []byte  // serialized command or event
    CRC32     uint32  // 資料完整性校驗
}

// Snapshot：定期將完整的 in-memory state 序列化
// 這樣 recovery 就不需要從頭重播所有 WAL
type Snapshot struct {
    Sequence   uint64           // 此 snapshot 對應的最後一個 WAL sequence
    OrderBooks map[string]*OrderBookState
    Positions  map[string]*PositionState
    Balances   map[string]*BalanceState
    Timestamp  int64
}

// Recovery 流程
func (e *Engine) Recover() error {
    snap, err := LoadLatestSnapshot()
    if err == nil {
        e.RestoreFromSnapshot(snap)
    }
    // 從 snapshot 之後的 sequence 開始重播
    entries := ReadWALFrom(snap.Sequence + 1)
    for _, entry := range entries {
        e.ReplayEntry(entry)  // 必須和正常處理邏輯 deterministic 一致
    }
}
```

3. 實作 snapshot 機制（定時 + 手動觸發）
4. 實作 recovery 流程，驗證恢復後狀態一致

### Phase 8：Performance & Production Patterns — 走向真實世界
**目標**：理解並實作真實交易系統的效能優化和設計模式

**核心設計模式總結**：

| 模式 | 用在哪裡 | 解決什麼問題 |
|------|---------|-------------|
| **Event Sourcing** | 整體架構 | 可審計、可重播、可恢復 |
| **Single Writer** | Matching Engine | 消除 lock contention，保證 determinism |
| **Append-Only Log** | WAL, Trade Log | Sequential write 比 random write 快 10-100x |
| **LMAX Disruptor Pattern** | Event bus | 無鎖的高吞吐 event 傳遞 |
| **Ring Buffer** | 內部通訊 | 固定大小、無 GC 壓力、cache friendly |
| **Snapshot + Replay** | Recovery | 快速恢復而不需重播全部歷史 |
| **Fan-Out / Pub-Sub** | Market Data | 一次計算，推送給多個消費者 |
| **Object Pool** | Order allocation | 減少 GC，穩定延遲 |
| **Sharding** | Multi-symbol | 每個 symbol 獨立的撮合引擎，水平擴展 |

**Go 特定的效能優化**：

```go
// 1. Object Pool — 避免頻繁 alloc/GC
var orderPool = sync.Pool{
    New: func() interface{} { return &Order{} },
}
// 取用：order := orderPool.Get().(*Order)
// 歸還：orderPool.Put(order)
// 為什麼重要？撮合引擎每秒可能建立/銷毀數十萬個 Order 物件
// GC pause 會直接影響 p99 latency

// 2. Lock-Free Ring Buffer
// Go 的 channel 底層有 mutex，在超高吞吐場景下會成為瓶頸
// Ring buffer 用 atomic CAS 實作 lock-free 的 producer-consumer
type RingBuffer struct {
    buf      []unsafe.Pointer
    capacity uint64
    mask     uint64  // capacity - 1 (capacity 必須是 2 的冪)
    head     uint64  // atomic
    tail     uint64  // atomic
}

// 3. 避免 map 在 hot path — 用 array index
// 如果 symbol 數量固定，可以 assign 每個 symbol 一個 int index
// 用 array[index] 取代 map[symbol]，避免 hash 計算和 bucket 遍歷

// 4. Protobuf / FlatBuffers 取代 JSON
// JSON encode/decode 在 hot path 上太慢
// FlatBuffers 甚至可以 zero-copy access
```

**Sharding 策略**：
```
每個 trading pair (BTC-USDT, ETH-USDT, ...) 擁有獨立的：
  - Matching Engine goroutine
  - Order Book
  - WAL file

共享的：
  - Balance Manager (需要跨 symbol 操作)
  - Risk Engine (cross margin 需要看所有倉位)
  - Market Data Publisher
```
